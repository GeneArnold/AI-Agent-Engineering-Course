# Contributing to AI Agent Engineering Course

First off, thank you for considering contributing to this course! Community contributions make this a better learning resource for everyone.

## ğŸ¯ How You Can Contribute

### 1. Report Issues

Found a bug, typo, or unclear explanation?

**Open an issue** with:
- Clear title (e.g., "Module 1: Typo in CONCEPTS.md")
- Description of the problem
- Location (file name, line number if possible)
- Suggested fix (optional)

[Open an issue](https://github.com/GeneArnold/AI-Agent-Engineering-Course/issues/new)

### 2. Suggest Improvements

Have an idea to make the course better?

**Open an issue** labeled "enhancement" with:
- What you'd like to see improved
- Why it would help learners
- How you envision it working

### 3. Submit Pull Requests

Want to contribute code or documentation directly?

#### For Bug Fixes

1. **Fork** the repo
2. **Create a branch:** `git checkout -b fix/module-1-typo`
3. **Make your changes**
4. **Test** thoroughly
5. **Commit** with clear message: `"Fix typo in Module 1 CONCEPTS.md"`
6. **Push** and create a Pull Request

#### For New Content

**Please open an issue FIRST** to discuss before creating a PR.

This ensures:
- Your work aligns with the course vision
- No duplicate efforts
- Faster review and merge

### 4. Share Your Experience

Completed the course? Share your story!

- Write a blog post
- Tweet about it (tag [@genearnold](https://twitter.com/genearnold))
- Post on LinkedIn
- Create video walkthroughs

**We'd love to feature your work!** Open an issue to let us know.

## ğŸ“‹ Contribution Guidelines

### Code Style

**Python:**
- Follow PEP 8
- Clear variable names (prefer clarity over brevity)
- Add comments explaining *why*, not just *what*
- Include docstrings for functions

**Markdown:**
- Use clear headings
- Break up long paragraphs
- Include code examples
- Link to relevant resources

### Content Principles

All contributions should align with the course philosophy:

âœ… **Concepts before code** - Explain the "why" first
âœ… **Simplicity over production** - Focus on clarity
âœ… **Deep and detailed** - Don't rush, ensure understanding
âœ… **Encourage experimentation** - Learning from mistakes is valuable

âŒ **Don't:**
- Add complex enterprise patterns
- Skip conceptual explanations
- Assume prior knowledge
- Use jargon without defining it

### Testing Changes

Before submitting a PR:

**For code:**
- Run the code yourself
- Test on a fresh environment (new venv)
- Verify logs/outputs match expected results

**For documentation:**
- Read it fresh (as if you're a learner)
- Check all links work
- Verify code examples run
- Test with Claude Code (if applicable)
